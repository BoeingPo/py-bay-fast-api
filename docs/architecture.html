Layers & the Dependency Rule

Outer rings depend inward on inner rings — never the reverse. Entities know nothing about FastAPI, SQLAlchemy, or boto3.

Frameworks & Drivers

FastAPI app wiring, DB engines/sessions, JWT & password hashing, settings

main.py — app, routers, startup
infrastructure/db/postgres.py — engine, SessionLocal, get_db
infrastructure/db/dynamo.py — get_table, serialize/deserialize
infrastructure/auth.py — JWT, password hashing, get_current_user
config.py — Settings (pydantic-settings)
Interface Adapters

Translate between the outside world (HTTP, DB drivers) and use cases

controllers/ health.py, users.py, concerts.py, bookings.py
UserRepository Postgres
ConcertRepository DynamoDB
BookingRepository DynamoDB
Use Cases (aka "service layer")

Application-specific business rules, orchestrate repositories

UserUseCases — register, authenticate
ConcertUseCases — list, get, create
BookingUseCases — book_seats, cancel_booking, get/list
Entities

Enterprise-wide rules — the only layer with zero outward dependencies

User — SQLAlchemy model (id, email, name, hashed_password, created_at)
Concert / Booking — plain dicts, no ORM entity (DynamoDB-native)

Dependency Rule — arrows point inward: Frameworks → Adapters → Use Cases → Entities, never the reverse.
